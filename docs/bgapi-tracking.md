# BGAPI tracking

## Overview

`bgapi` is the FreeSWITCH command mode for long-running operations. Unlike `api`, which blocks the ESL connection until a reply arrives, `bgapi` returns immediately with a job UUID. FreeSWITCH executes the command in a background thread and emits a `BACKGROUND_JOB` event when the job completes.

`esl-react` implements a tracked dispatch model that bridges the synchronous job UUID return with an async promise that resolves on completion.

---

## Dispatch

```php
$handle = $client->bgapi('originate', 'sofia/internal/1000 &echo');
```

`bgapi()` returns a `BgapiJobHandle` **immediately** — before the command has been written to the socket, before FreeSWITCH has received it, and before any completion event exists.

The handle is valid from the moment `bgapi()` returns. Its promise will be resolved or rejected asynchronously.

`bgapi()` does NOT return a `PromiseInterface` directly. It returns a `BgapiJobHandle`. The handle's promise is accessed via `$handle->promise()`.

---

## BgapiJobHandle

```php
$handle->jobUuid();    // string — the Job-UUID FreeSWITCH will use
$handle->promise();    // PromiseInterface<BackgroundJobEvent>
```

The Job-UUID is generated by the runtime before the command is sent. FreeSWITCH honors the Job-UUID specified in the command header. This allows the runtime to pre-register the pending job before the command is written.

The promise resolves with a `BackgroundJobEvent` when the corresponding completion event arrives. The `BackgroundJobEvent` carries the job output in its body.

---

## Completion matching

When a `BACKGROUND_JOB` event arrives from FreeSWITCH:

1. `InboundMessageRouter` classifies it as a bgapi completion (not a typed event for listener delivery).
2. `BgapiCompletionMatcher` reads the `Job-UUID` header from the envelope.
3. It looks up the matching `PendingBgapiJob` in `BgapiJobTracker` by UUID.
4. If found, the job's promise is resolved with the event and the job is removed from the tracker.
5. If no matching job is found, the completion event is discarded.

---

## Timeout behavior

Bgapi completions are subject to two separate timeouts, configured via `CommandTimeoutConfig`:

### bgapiAckTimeoutMs

The time allowed for FreeSWITCH to acknowledge the `bgapi` command with its initial reply (the short reply that confirms the command was received and the job UUID). If this reply is not received within `bgapiAckTimeoutMs`, the handle's promise is rejected with `CommandTimeoutException`.

This is distinct from the job completion. An ack timeout means FreeSWITCH never accepted the command.

### bgapiCompletionTimeoutMs

The time allowed for the `BACKGROUND_JOB` completion event to arrive after the ack. If no completion arrives within this window, the handle's promise is rejected with `CommandTimeoutException`.

This timeout begins when the ack is received, not when `bgapi()` is called.

### Late completions

If a `BACKGROUND_JOB` event arrives for a job whose promise has already been rejected due to timeout, it is **silently discarded**. The job UUID is no longer in `BgapiJobTracker`, so no lookup matches and no action is taken.

---

## Orphaned jobs

An orphaned job is one where the completion event never arrives (FreeSWITCH lost the job internally, the originate failed without emitting `BACKGROUND_JOB`, or FreeSWITCH was restarted mid-execution).

Orphaned jobs are cleaned up by the completion timeout. After `bgapiCompletionTimeoutMs` elapses, the promise rejects with `CommandTimeoutException` and the job is removed from the tracker.

Set `bgapiCompletionTimeoutMs` to a value longer than the longest expected job duration. For originate commands, this may be 60+ seconds. For shorter commands, shorter values are appropriate.

---

## Reconnect behavior

`bgapi` jobs survive reconnect. When the connection drops:

- Pending bgapi jobs are NOT rejected (unlike inflight `api` commands).
- Their promises remain open.
- `BgapiJobTracker` retains all pending job UUIDs.

If FreeSWITCH was processing the job before the disconnect, it may still emit the `BACKGROUND_JOB` completion event after the ESL connection is reestablished. When that event arrives on the new connection, `BgapiCompletionMatcher` matches it by UUID and resolves the handle's promise normally.

If FreeSWITCH also lost the job during the restart, no completion will ever arrive, and the job will time out via `bgapiCompletionTimeoutMs`.

This design means pending bgapi promises are resolved correctly whenever possible after reconnect, without the caller needing to reissue the command.

---

## Pending bgapi count

`HealthSnapshot::$pendingBgapiJobCount` reflects the current number of tracked bgapi jobs. Under normal operation this should be low and bounded. A growing count indicates either very long-running jobs, a high dispatch rate, or orphaned jobs accumulating before timeout.

The `BackpressureConfig::$maxPendingBgapiJobs` threshold can be used to reject new bgapi dispatches when the pending count exceeds a configured limit. Rejection throws `BackpressureException`.

---

## Usage example

```php
$handle = $client->bgapi('originate', '{origination_uuid=my-call-uuid}sofia/internal/1000 &echo');

echo "Dispatched job: " . $handle->jobUuid() . "\n";

$handle->promise()->then(
    function (\Apntalk\EslCore\Model\Event\BackgroundJobEvent $event) {
        echo "Job completed: " . $event->getBody() . "\n";
    },
    function (\Throwable $e) {
        if ($e instanceof \Apntalk\EslReact\Exceptions\CommandTimeoutException) {
            echo "Job timed out.\n";
        } elseif ($e instanceof \Apntalk\EslReact\Exceptions\ConnectionLostException) {
            // This should not happen for bgapi (bgapi survives reconnect),
            // but handle it if RetryPolicy::disabled() and connection is gone permanently.
            echo "Connection lost permanently.\n";
        }
    }
);
```
